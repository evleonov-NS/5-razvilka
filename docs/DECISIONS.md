# DECISIONS.md — принятые архитектурные решения

Зафиксированные решения для «Развилка» MVP. Не пересматривать без явной причины.

---

## ADR-001: Next.js App Router + TypeScript

**Решение:** Next.js с App Router и TypeScript.

**Причина:** единый fullstack-фреймворк, Server Components для чтения БД, Route Handlers для мутаций, нативный деплой на Vercel.

**Альтернативы отклонены:** отдельный SPA + API (лишняя сложность для MVP).

---

## ADR-002: Prisma 6 + Neon PostgreSQL

**Решение:** Prisma ORM, Neon как managed PostgreSQL.

**Причина:** типобезопасность, миграции, `url` (pooled) + `directUrl` (миграции) — стандарт для serverless.

**Примечание:** PrismaClient — синглтон в `lib/prisma.ts`.

---

## ADR-003: Auth.js v5 + Google OAuth (пересмотр 2026-07-05)

**Было:** собственная auth: bcrypt + JWT в httpOnly cookie, без NextAuth.

**Решение:** [Auth.js v5](https://authjs.dev/) (`next-auth`) + **только Google OAuth**. Email/пароль — вне scope MVP.

**Причина:**
- готовая библиотека: OAuth flow, callback, CSRF, session cookie;
- Prisma Adapter — пользователь и сессии в своей Neon БД;
- сессия в **httpOnly cookie** (database strategy), не localStorage;
- один провайдер — минимальная сложность UI (кнопка «Войти через Google»).

**Схема Prisma:** `User` (без `passwordHash`) + `Account` + `Session` + `VerificationToken` (стандарт adapter).

**API для приложения:** `auth()` / `getCurrentUser()` / `requireUser()` — обёртки над сессией Auth.js. Route Handler: `app/api/auth/[...nextauth]/route.ts`.

**Env:** `AUTH_SECRET`, `AUTH_URL`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`.

**Отклонено:** email+пароль (лишний UX и bcrypt); Better Auth (выбран Auth.js по запросу).

## ADR-004: Route Handlers для мутаций, Server Components для чтения

**Решение:** мутации — только `app/api/.../route.ts`; чтение — Server Components напрямую из Prisma.

**Причина:** явное разделение, без Server Actions (зафиксировано в ТЗ).

**Runtime:** Node.js (не Edge) — Prisma + pg несовместим с Edge.

---

## ADR-005: OpenAI-совместимый LLM-клиент

**Решение:** npm-пакет `openai` с настраиваемым `OPENAI_BASE_URL`. Реализация: `lib/llm.ts` (`chatCompletion`, `isLlmConfigured`).

**Env:** `OPENAI_API_KEY` (обязателен для вызова), `OPENAI_BASE_URL` (опционально), `LLM_MODEL` (обязателен для вызова).

**Причина:**
- prod на Vercel — прямой OpenAI;
- локально из РФ — агрегатор/прокси без смены кода;
- один интерфейс для разных провайдеров.

**Правило:** вызовы LLM только на сервере, ключ из `process.env`. Не импортировать `lib/llm.ts` в Client Components.

**Примечание (2026-07-23):** peerOptional `zod@^3` у `openai@5` конфликтует с Zod 4 проекта — установка через `npm install openai --legacy-peer-deps` (Zod helpers пакета не используем).

## ADR-006: Zod для ввода и ответов LLM

**Решение:** все входы API и все JSON-ответы LLM валидируются через Zod.

**Pipeline:** raw text → `lib/json.ts` (срез markdown) → `JSON.parse` → Zod schema.

**При невалидном ответе:** не сохранять в БД, вернуть 502/422, залогировать raw (без секретов).

## ADR-007: Enum в ВЕРХНЕМ регистре + Zod preprocess

**Решение:** промпты просят LOW/MEDIUM/HIGH, OPTIMISTIC/BASE/PESSIMISTIC; Zod нормализует регистр через `z.preprocess` + `toUpperCase()`.

**Причина:** совпадение с Prisma enum без отдельного маппера.

## ADR-008: Дерево развилок — отдельная генерация

**Решение:** `POST /api/decisions/[id]/tree`, не вместе со сценариями.

**Причина:** разделение LLM-вызовов (скорость, стоимость, UX: пользователь решает, нужно ли дерево).

**Хранение:** `Decision.tree Json?`.

## ADR-013: `/` — лендинг или журнал в зависимости от сессии (2026-07-20)

**Решение:** `app/page.tsx` через `getCurrentUser()`: нет сессии → продуктовый лендинг (`LandingPage`); есть сессия → `redirect("/cabinet")` (журнал решений).

**Причина:**
- гостю нужен оффер и объяснение продукта на корневом URL;
- авторизованному — сразу рабочий журнал, без повторного маркетингового экрана;
- кабинет уже является журналом; дублировать разметку на `/` нет смысла.

**Отклонено:** отдельный `/landing` (лишний hop с корня); журнал прямо в `page.tsx` рядом с лендингом (два крупных UI в одном файле).

## ADR-014: Семантические CSS-токены вместо `dark:` префиксов (2026-07-20)

**Решение:** палитра через CSS-переменные (`--bg`, `--surface`, `--text`, `--accent`…) на `:root`/`.light` и `.dark`; в разметке только семантические классы Tailwind (`bg-surface`, `text-text-muted`). Класс темы на `<html>` (`darkMode: 'class'`).

**Причина:**
- один компонент — один набор классов; нет дублирования `bg-white dark:bg-black`;
- смена темы = смена значений переменных, без переписывания утилит;
- `color-scheme` чинит нативные скроллбары и поля ввода.

**Отклонено:** повсеместные `dark:` префиксы (шум и риск забыть состояние).

## ADR-015: Раздельные `--accent` и `--accent-ink` (2026-07-20)

**Решение:** `--accent` — яркая янтарная заливка (кнопки, узлы SVG); `--accent-ink` — цвет акцентного текста (ссылки, номера, метки); `--accent-contrast` — текст поверх заливки.

**Причина:** один янтарь не может одновременно быть читаемым текстом на кремовом фоне (контраст ~1.9:1 — провал) и живым брендовым цветом кнопки. В светлой теме `--accent-ink` темнеет (`#8f5c15`); в тёмной совпадает с `--accent`. Текст на кнопке — почти чёрный (`accent-contrast`), не белый.

**Отклонено:** затемнять заливку ради белого текста (мутная «коричневая» CTA).

## ADR-016: backdrop-filter только у шапки — утилита читаемости (2026-07-20)

**Решение:** `backdrop-filter` / `-webkit-backdrop-filter` применяется исключительно к sticky-шапке (`SiteHeader`) с полупрозрачным `rgb(var(--bg-rgb) / 0.72)`, blur ≤16px. Декоративный glassmorphism (окантовка «стекла», свечение, плавающая капсула, цветные тени) остаётся под запретом.

**Причина:** при прокрутке шапка должна оставаться читаемой поверх контента; лёгкий blur — средство читаемости, не визуальный «AI-эффект». Без поддержки blur — почти непрозрачный фон (`/0.97`).

**Отклонено:** fixed-шапка (скачок контента); анимация самого `backdrop-filter` (рывки); glassmorphism как стиль продукта.

---

## ADR-017: Пустой журнал — пресеты и призрачное превью (2026-07-20)

**Решение:** при нуле решений в журнале показывать приглашение «С чего начнём?» со стартовыми чипсами-пресетами (`lib/presets.ts` → `/decisions/new?preset=…`) и неинтерактивным призрачным превью трёх сценариев; без пунктирной «пустой» рамки.

**Причина:** констатация «у вас нет решений» повышает барьер первого действия. Пресет даёт правимый черновик (title + context), а превью показывает ценность разбора до первого вызова LLM — человек правит готовое, а не пишет с нуля.

**Отклонено:** одна CTA-кнопка в dashed-box; выдуманные проценты/цифры в превью; цветная оценочность LOW/MEDIUM/HIGH (только насыщенность акцента).

## ADR-018: «Сообщество» вне навигации кабинета на MVP (2026-07-20)

**Решение:** пункт «Сообщество» убран из сайдбара кабинета; маршрут `/explore` и механики `isPublic`/лайков сохранены. Навигация MVP: Журнал · Открытые · Решённые. Настройки — профиль/тема/выход (без заглушки), доступ с блока профиля.

**Причина:** в PROJECT.md §2 социальная лента отложена; пустой пункт «Скоро…» и акцент на чужих разборах отвлекают от первого собственного разбора — главной цели кабинета.

**Отклонено:** оставлять «Сообщество» и «Настройки: Скоро…» в сайдбаре как сигнал незавершённости.

## ADR-019: Сайдбар на всех экранах кабинета (2026-07-20)

**Решение:** `CabinetShell` (боковая панель) оборачивает не только `/cabinet/*`, но и `/decisions/*` и `/demo`. Узкая колонка формы (`max-w-2xl`) даёт фокус на вводе; навигация не исчезает.

**Причина:** снятие сайдбара на форме вызывало скачок вёрстки и потерю ориентира при Журнал → форма → назад. Ощущение «режима фокуса» достигается шириной контента, а не удалением chrome.

**Отклонено:** отдельный layout без сайдбара для `/decisions/new` и экрана генерации.

## ADR-020: Статический `/demo` как рычаг активации (2026-07-20)

**Решение:** страница `/demo` показывает полный пример разбора («Сменить работу…») из `lib/demo-decision.ts` — сценарии, pre-mortem, дерево, пример ревью. Без LLM и без записи в БД; в журнал не попадает. Ссылка из пустого журнала: «Посмотреть пример целиком».

**Причина:** призрачное превью показывает только три сценария; полный продукт (pre-mortem, дерево, ревью) человек не видит до первого разбора. Демо снимает неопределённость «а что я получу?» без ожидания LLM.

**Отклонено:** сохранять демо как реальное Decision в БД; показывать только сценарии на `/demo`.

## ADR-021: Sticky-сайдбар без overflow у предков (2026-07-20)

**Решение:** боковая панель — `sticky top-0 self-start h-[100dvh] overflow-y-auto`, ширина `w-56`, `shrink-0`. Корень оболочки — `flex min-h-screen` **без** `overflow-x-hidden` / `overflow-hidden`. Горизонтальное переполнение лечится у виновника (`min-w-0` на колонке контента). На md у панели `transform-none` (mobile-drawer transform иначе ломает sticky).

**Причина:**
- любой `overflow` ≠ `visible` у предка делает его scroll-контейнером — sticky липнет к нему и уезжает с длинной страницей;
- без `self-start` flex растягивает панель по высоте контента — прилипать некуда;
- app-shell (`h-screen` + отдельный скролл контента) отклонены: проблемы 100vh на мобильных, вложенные скроллбары, ломается восстановление позиции при «назад».

**Отклонено:** `overflow-x-hidden` на корне кабинета; app-shell с внутренним скроллом контента; `position: fixed` панели на десктопе.

## ADR-022: Multi-provider LLM + квоты + стоимость (2026-07-23)

**Решение:**
- Платформа по умолчанию — **DeepSeek** (`DEEPSEEK_API_KEY`, `LLM_DEFAULT_PROVIDER=DEEPSEEK`).
- В `/cabinet/settings` пользователь выбирает провайдера (DeepSeek / Qwen / OpenAI), модель и свой API-ключ.
- Ключ хранится в `User.llmApiKeyEnc` (AES-256-GCM от `AUTH_SECRET`), в UI только маска.
- Квоты: `OWNER_EMAIL` (по умолчанию `evleonov79@gmail.com`) — безлимит на платформенном ключе; остальные — `FREE_PLATFORM_CREDITS=1`, затем свой ключ.
- Стоимость: таблица `LlmUsage` (токены + оценка USD по прайсу каталога `lib/llm/providers.ts`).

**Причина:** свой BYOK снимает стоимость с сервиса; один тестовый разбор снижает барьер входа.

**Связанные файлы:** `lib/llm/*`, `GET|PUT /api/settings/llm`, миграция `user_llm_settings`.
